# CLAUDE.md — Example (Magento 2 + React/Vite)

> **This is an example CLAUDE.md** from a Magento 2 + React/Vite project. It demonstrates the slim structure that the `.claude/` toolchain expects. Convention-specific rules (code standards, commit format, testing, language conventions) live in `.claude/rules/` — see the corresponding rules files in this directory. When you run `/setup-project`, a project-specific CLAUDE.md and rules files will be generated.

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Magento 2 Enterprise Edition (v2.4.7) ecommerce platform. The frontend combines Magento's Luma-based theme (`app/design/frontend/VendorName/theme-name/`) with React widgets compiled via Vite (`app/code/VendorName/React/`).

- **Vendor namespace:** `VendorName` — all custom modules live under `app/code/VendorName/`

## Commands

### React / Frontend (Node)

Run all from the repo root:

```bash
yarn dev           # Vite dev server (requires local HTTPS certs)
yarn build         # Production build -> app/code/VendorName/React/view/frontend/web/
yarn lint:check    # ESLint check on React source
yarn lint:fix      # ESLint auto-fix
yarn check-types   # TypeScript type check (no emit)
```

- Node version: **v20** (see `.nvmrc`)
- Package manager: **Yarn v4** (`yarn.lock`)

### Magento CLI

All Magento CLI commands use the `<your-cli-wrapper>` wrapper — **never** use `bin/magento` directly:

```bash
<your-cli-wrapper> magento setup:upgrade        # Run after adding new modules
<your-cli-wrapper> magento setup:di:compile     # Run after DI changes (di.xml, plugins, preferences)
<your-cli-wrapper> magento cache:flush          # Flush all caches
```

### PHP / Magento (Composer)

Pre-commit hooks run these automatically via `<your-cli-wrapper>`. Run manually using:

```bash
<your-cli-wrapper> composer run check-style   # PHPCS (Magento2 standard) on staged .php/.phtml files
<your-cli-wrapper> composer run fix-style     # PHPCBF auto-fix on staged files
<your-cli-wrapper> composer run analyse       # PHPStan on staged files
```

## Architecture

### React + Vite Integration

React source lives entirely in one module:

```
app/code/VendorName/React/view/frontend/
  src/app/
    widgets/        <- Vite entry points (auto-discovered; one file = one bundle)
    components/     <- Shared React components
    api/            <- API layer (GraphQL client, operations, providers)
    hooks/          <- Custom hooks
    context/        <- React context providers
    utils/          <- Utility functions
    types/          <- TypeScript types
    constants/      <- Shared constants
  web/              <- After build: build output (NOT committed; generated by `yarn build`)
    js/             <- After build: <widget-name>.js per entry
    css/            <- After build: react-bundle.css (single CSS bundle)
```

**To add a new widget:** create a `.tsx` file in `widgets/` — Vite auto-discovers it.

**Mounting convention:** Widgets use `data-react-widget` attribute lookup (`document.querySelectorAll('[data-react-widget="name"]')`) to support multiple instances on one page. Data passes from Magento to React via HTML data attributes on the mount `<div>`.

**Script loading:** The `ReactComponent.php` Block class handles loading — dev mode points to the Vite dev server, production points to compiled assets (e.g. `VendorName_React::js/my-widget.js`). React widgets are **ES modules** (not RequireJS). CSS is a single bundled `react-bundle.css`.

**GraphQL access:** Widgets use a project config singleton (not React Context). The GraphQL endpoint URL is built from environment variables.

Path alias `@/*` resolves to `src/app/*`.

### Magento Theme

```
app/design/frontend/VendorName/theme-name/
  web/css/source/   <- LESS overrides
  web/js/           <- RequireJS modules
  Magento_*/        <- Per-module layout XML and template overrides
```

Theme inherits from Luma. Prefer `referenceBlock`/`referenceContainer` in layout XML over copy-pasting.

### Custom Magento Modules

All under `app/code/VendorName/`:

| Module     | Purpose                                               |
| ---------- | ----------------------------------------------------- |
| `React`    | Hosts all React/Vite frontend source and build output |
| `Catalog`  | Product catalog customizations                        |
| `Customer` | Customer account customizations                       |
| `Theme`    | Theme-level module utilities                          |

Standard Magento module structure: `etc/`, `Block/`, `Model/`, `Plugin/`, `view/`, `registration.php`.

**Plugin naming convention:** `<prefix>_<module>_<action>` — e.g. `acme_catalog_add_custom_attribute`. Use this prefix for all plugin names in `di.xml`.

### Key Dependencies

- **Apollo Client** — GraphQL queries to Magento's GraphQL API
- **Tailwind CSS v4** — Utility-first styling in React components; screen-reader-only text uses the `sr-only` class
- **@headlessui/react** — Accessible UI primitives
- **Zod** — Runtime validation

## Documentation (`docs/`)

The `docs/` folder is an active part of the development workflow, not just a reference archive. Agents, skills, and commands read from and write to it during normal feature work.

```
docs/
  requirements/   <- Jira tickets and specs (fetched via scripts/); input for feature-planner agent
  plans/          <- Implementation plans produced by feature-planner; input for feature-implementer
  features/       <- Architecture documents for completed features (Mermaid diagrams, data flows)
  manuals/        <- Workflow guides for using Claude Code tools effectively
  scripts/        <- Workflow utility scripts (Jira fetcher, etc.)
```

### How each folder is used

**`docs/requirements/`** — Jira ticket content (description, comments, mockup images) fetched via `docs/scripts/fetch-jira-ticket.sh`. The `feature-planner` agent reads these to understand scope — including attached images for UI placement. To fetch a ticket: `./docs/scripts/fetch-jira-ticket.sh <TICKET-ID>` (reads credentials from `.env.development` — see `.env.development.example`)

**`docs/plans/`** — Save `feature-planner` output here as `PROJ-XXX-feature-name.md` before implementation begins. The `feature-implementer` agent takes a plan file as its input. Plans are living documents — update them if scope changes during implementation.

**`docs/features/`** — **Mandatory for complex features.** After implementing a multi-layer feature, create an architecture document here before creating the PR. Run `/document PROJ-XXX` to generate it, then commit as the final commit on the branch. Without this document, future developers and AI agents cannot understand the feature's design without re-reading every file. See existing files in this folder for reference examples. The `/commit` skill prompts for this after commits are created.

**`docs/manuals/`** — Structured AI tooling docs by intent (`getting-started/`, `workflows/`, `playbooks/`, `reference/`, `concepts/`). Start from `docs/manuals/README.md`. Do not edit these manually — they are maintained alongside the `.claude/` tool definitions.

### When to consult `docs/` before starting work

- **Picking up an existing feature** -> read `docs/features/<PROJ-XXX>.md` for architecture context before touching any code
- **Starting a new feature** -> check `docs/plans/` for an existing plan before asking `feature-planner` to re-plan
- **Unsure which tool to use** -> read `docs/manuals/reference/ai-tools-reference.md`
- **New to the repo** -> start with `docs/manuals/getting-started/onboarding.md`
